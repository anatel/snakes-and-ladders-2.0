# Multiplayer Online Play Plan

Status: draft
Owner: Anat Eliahu
Last updated: 2026-07-17

## Goal
Add online multiplayer to the Snakes and Ladders game: a player enters a display name, creates or joins a game (up to 4 players), takes turns with a 1-minute per-turn timeout that auto-removes inactive players, and the last remaining player wins — per the updated `.doc/product-definition.md`.

## Scope
In scope:
- Entry flow: player enters a display name and chooses a mode — player vs. computer, or online multiplayer.
- Multiplayer lobby: for multiplayer, the player creates a new, named game or picks one to join from a list of open (waiting) games showing each game's name and current player count.
- Game session: up to 4 players per game; players can join while the game is in a pre-start "waiting" state.
- Real-time sync of game state (board positions, whose turn it is, dice results) across all connected players in a session.
- Turn timer: 1-minute countdown per turn; if the current player hasn't rolled within that window, they're automatically removed from the game and the turn passes to the next active player.
- Reconnection: if a player leaves the page (navigates away, closes the tab, loses connection) and returns within that same 1-minute window, they resume their seat and the game continues as if they hadn't left; the countdown only results in removal if it fully expires.
- Win-by-forfeit: if removals leave exactly one player remaining, that player is declared the winner immediately.
- Reuses existing board/dice/turn/win-detection game logic (`app/src/game/gameLogic.ts`, `app/src/game/board.ts`) rather than re-implementing it.
- A minimal backend process to hold authoritative game state, broadcast updates, and enforce turn timeouts server-side (client-only timers aren't trustworthy for enforcement across multiple independent clients).

Out of scope (deferred, per product definition):
- Accounts, persistent stats, leaderboards, chat, spectator mode.
- The existing player-vs-computer mode (unaffected, remains as-is).
- Rejoining a game once the 1-minute window has fully expired and the player has been removed (that removal is final for the session).
- Public hosting/deployment of the backend (local/dev only for this plan, consistent with plan 001's stance).

## Assumptions
- A lightweight Node.js WebSocket server (e.g. `ws` or `socket.io`) with in-memory game-session state is sufficient — no persistent database needed, since games/players aren't stored beyond the session (matches "no accounts/persistent stats" in product scope and the "dependency-light" prioritization rule).
- The 1-minute turn timer is enforced server-side (the server, not any single client, is authoritative on timeout) so the game stays correct even if a player's tab is closed or their connection drops.
- A player is identified only by the display name they type when creating/joining a game (no login), matching the updated product-definition assumption.
- Session IDs are short-lived and generated per game, not persisted after all players leave.
- A returning client needs some way to prove it's the same player reclaiming a seat rather than a new joiner (see Open Question 6) — this is a client-held token, not an account.

## Open Questions
1. ~~Backend hosting approach~~ — **Decided: standalone Node + WebSocket server (`ws`/`socket.io`) run alongside the Vite app in dev (e.g. a new `server/` directory), in-memory state, no DB.**
2. ~~Late joining by a new player~~ — **Decided: waiting-room only. A new player can only join while the game is `waiting`; once started, the 4 slots are locked for the session (minus reconnects of the original players).**
3. ~~Turn order~~ — **Decided: join order (the order players joined the game).**
4. ~~Minimum players to start~~ — **Decided: require at least 2 players before the creator can start the game; the "start game" action is disabled until then.**
5. ~~Removed-player fate mid-board~~ — **Decided: remove the piece from the board and mark the player "left" in the player list.**
6. ~~Reconnect identity~~ — **Decided: the server issues each player a per-session token on join; the client stores `{gameId, playerToken}` in `localStorage`, and the game has a shareable URL (`/game/:gameId`). Returning to that URL within the 1-minute window lets the client silently re-authenticate and resume its seat, canceling the removal countdown.**
7. ~~Open-games list access~~ — **Decided: fully public list, no access control. Any open (waiting) game is visible to anyone in multiplayer mode; the game name is just a convenience for a friend to pick the right one. Private/invite-only games are a future enhancement, out of scope here.**

## Steps
1. Design the session/message protocol: game states (`waiting`, `in-progress`, `finished`), player states (`active`, `disconnected`, `left`), and WebSocket event/message shapes (create game, join game, start game, roll, reconnect, turn-timeout, player-left, game-state-sync, open-games-list).
2. Scaffold a minimal backend (`server/`): Node + TypeScript WebSocket server, in-memory `Map` of game sessions keyed by game ID.
3. Implement session management: create game (creator supplies a game name, generates a game ID, first player = creator), join game (adds player if a slot is free and the game is `waiting`), list open games (returns `waiting` games with name + current/max player count), start game (creator-triggered, requires the minimum player count from Open Question 4).
4. Implement the server-authoritative turn loop: track the current player, start a 1-minute server-side timer per turn, broadcast state to all connected clients on every change.
5. Implement reconnect + turn-timeout handling: on disconnect, mark the player `disconnected` but keep their seat and let the existing per-turn timer keep running; if their client reconnects with a valid token (Open Question 6) before the timer expires, mark them `active` again and cancel the pending removal; if the timer fully expires first, mark the player `left`, remove their piece, and advance the turn to the next active player — if only one active player remains, end the game with that player as winner.
6. Wire dice rolls and movement through the server: client sends "roll," server generates the result (server-authoritative randomness prevents client tampering), runs the existing move/snake/ladder resolution logic, and broadcasts the outcome.
7. Build the client entry/mode-selection screen: name entry, then a choice between "vs. computer" and "online multiplayer."
8. Build the client multiplayer lobby UI: "create game" (with a name field) and a list of open games (name + player count) to join; a waiting room showing joined players with a "start game" action for the creator.
9. Build the client multiplayer game view: reuse the `Board`/`Dice`/`Piece` components, add a turn-countdown indicator and a player-left notification (via `sonner`, per `.rule/ui-rules.md`).
10. Add a `useMultiplayerGame` hook (mirroring the existing `useGame.ts` pattern) that manages the WebSocket connection, persists `{gameId, playerToken}` to `localStorage`, and on mount attempts to reconnect using a stored token if one matches the current game URL.
11. Implement the win/forfeit end screen: distinguish "won by reaching square 100" vs. "won because everyone else left," both with a "Play again" / "back to lobby" action.
12. Add tests per `.rule/testing-rules.md`: server-side session/turn/timeout/reconnect logic (unit tests with mocked/frozen timers) and client hook behavior with a mocked WebSocket.
13. Manual pass: run 2-4 browser tabs/clients against the local server, play a full game to completion; verify a player who closes and reopens a tab within 1 minute resumes their seat, and one who stays away past 1 minute is removed with the win correctly handed to the last remaining player.

## Validation
- Automated: unit tests for session creation/join/start/list, turn-timeout removal, reconnect-within-window, and win-by-last-player-standing logic pass; timers mocked/frozen per `.rule/testing-rules.md`.
- Manual: open multiple browser tabs as separate players, create a named game, confirm it shows up correctly in the open-games list, join and play through dice rolls/snakes/ladders to a normal win. Separately: close a tab mid-game and reopen the same game URL within 1 minute, confirming the player resumes their seat and turn; let another player stay away past 1 minute and confirm they're auto-removed and the game resolves correctly when only one player remains.
- Run the `verify` skill against the running app (client + server) before calling this done.

## Risks
- Server-side timers for many concurrent games could accumulate if sessions aren't cleaned up — mitigate by clearing a session's timers/state once a game finishes or all players leave.
- A reconnect racing the exact moment its timer expires could either wrongly restore a just-removed player or wrongly remove a just-reconnected one — mitigate by making removal and reconnect mutually exclusive on the server (single authoritative state transition per player, no client-side guessing).
- Client and server can drift out of sync if any game-state mutation happens client-side — mitigate by keeping the server fully authoritative (client only sends intents like "roll," never computes results itself).
- Running a WebSocket server alongside the existing static Vite app is a real architecture change (no longer purely client-only) — mitigate by keeping the server minimal and documenting it in `.doc/architecture.md` once implemented.

## Rollout Order
1. Land the backend session/turn/timeout logic first, validated by tests, with no UI (protocol-level only).
2. Add the client lobby and multiplayer game view wired to the server.
3. Add polish (turn-countdown UI, player-left toasts) last.
4. Ship as a single release, running locally (client `npm run dev` + server dev process); no phased user rollout and no public hosting in this plan, consistent with plan 001.

## Rollback
- Revert the branch/commit. The multiplayer feature is additive (new `server/` process plus new client routes/components) and doesn't change the existing player-vs-computer mode, so rollback doesn't affect the already-shipped MVP.
